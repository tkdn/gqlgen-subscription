# GraphQL Subscription 疎通検証プロジェクト実装計画

## Context

GraphQL subscriptionによるリアルタイム通知の疎通を検証したい。バックエンドはGo(gqlgen)、フロントエンドはAngularという構成で、ジョブ（概念のみで実体は重要でない）の状態変化を、Redis Pub/Sub経由でバックエンドが検知し、SSE(Server-Sent Events)トランスポートでフロントエンドへ配信する。

検証の主眼はsubscription配信の疎通確認そのものであり、以下の点を優先する。
- ジョブの状態を検証用mutationで自由に変更できること
- ジョブはRedisのTTLで揮発してよいこと（永続化不要）
- ユーザーごとにジョブが分離され、他ユーザーのジョブは見えないこと（擬似認証でよい）
- クライアント側の差分マージロジックまで検証範囲に含めないよう、配信は全件スナップショット方式とする

技術選定の過程で、gqlgen v0.17.93のSSE実装（`event: next` / `event: complete`形式）とフロントエンドで使う`graphql-sse`クライアントのデフォルト動作（Distinct Connections Mode）が、実際のソースコードレベルで互換性があることを確認済み。Docker Desktop側のWSL統合も有効化済みで `docker compose` が使用可能（v5.3.0）。

## 確定した仕様

### ジョブの状態モデル
```graphql
enum JobState {
  PENDING
  ANALYZING
  GENERATING
  COMPLETED
  FAILED
}
```
状態遷移に制約はない。検証用mutationで任意の状態に自由に変更できる。

### ユーザー分離（擬似認証）
- HTTPミドルウェアで固定値のUserIDを`context.Context`に注入する。本来はここでトークン検証を行うべき、という趣旨のコメントを残す。
- GraphQLスキーマの引数に`userId`は持たせない。resolverはすべて`ctx`からUserIDを取り出してjobstore層に渡す。

### Redisデータ設計
```
user:<userID>:jobs   → Set { "job-1", "job-2", ... }        # そのユーザーのジョブ名索引（TTLなし）
job:<userID>:<name>  → Hash { status: "ANALYZING" }          # 個別ジョブの状態（TTL付き、5分）
```
- `job:<userID>:<name>`をキー命名の起点にすることで`SCAN`を回避し、Setによる索引から直接メンバーを取得する。
- TTL失効によりHash実体が消えてもSetのメンバーは自動的には消えないため、**`List(userID)`実行時に読み取り時ガベージコレクションを行う**: Setの各メンバーについて`HGETALL job:<userID>:<name>`が空を返したら、その場で`SREM user:<userID>:jobs <name>`する。
- TTL定数: 5分。`Create`/`UpdateStatus`のたびに`EXPIRE`で再設定する。

### Pub/Sub設計
- チャンネル名は`job:updates:<userID>`とUserIDごとに分離する（Redisサーバー自体は1つ、チャンネルを分けるだけ）。
- `Hub`はプロセス内に1つ。UserIDごとにRedis Subscribe接続を遅延生成し（初回subscription時）、以後は同一UserIDのsubscriptionリクエストをその接続にfan-outする。誰も購読しなくなったUserIDの接続は破棄する。
- Publishは`jobstore.Store`の`Create`/`UpdateStatus`実行成功直後、同一メソッド内で行う（呼び出し漏れ防止）。メッセージpayloadは空でよく、受信はトリガーとしてのみ使う（受信したら`List(userID)`を呼び直してスナップショットを再取得する）。

### GraphQLスキーマ全体
```graphql
enum JobState {
  PENDING
  ANALYZING
  GENERATING
  COMPLETED
  FAILED
}

type Job {
  name: String!
  status: JobState!
}

type Query {
  jobs: [Job!]!
}

type Mutation {
  createJob(name: String!): Job!
  updateJobStatus(name: String!, status: JobState!): Job!
}

type Subscription {
  jobStatuses: [Job!]!
}
```
- `Query.jobs`は全件取得。curlでのE2E確認や、フロントエンドがsubscription接続前に初期表示するためにも使える。

### トランスポート
- gqlgen `transport.SSE{KeepAlivePingInterval: 10 * time.Second}`を他のtransport（Options/GET/POST）より先に追加する。WebSocketは使わない。

### フロントエンド技術選定
- Angular CLI最新版（22.0.5）、パッケージマネージャーはpnpm。
- apollo-angular、graphql-request、GraphQL Code Generatorは導入しない。
- Subscription: `graphql-sse`パッケージ（デフォルトのDistinct Connections Modeを使用、gqlgenのSSE実装と互換性確認済み）。
- Query/Mutation: Angular `HttpClient` + `provideHttpClient(withFetch())`で自前のfetchベース実装。mswとの相性を考慮しfetchバックエンドを明示的に使う。
- 型定義（`Job`/`JobState`）は手書きのTypeScript型とする。

### テスト方針
- バックエンド: resolver単体テスト（jobstore/hubをインターフェース経由でモック）、jobstore/hub統合テスト（miniredis使用）、SSEエンドポイント全体のhttptest E2Eテスト。
- フロントエンド: `@testing-library/angular`でコンポーネントをレンダリングしてテスト。Query/MutationのHTTPリクエストは`msw`でモック。Subscription（SSE）はmswでのインターセプトが困難なため、`graphql-subscription.service`自体をテスト用スタブに差し替える。

## バックエンド実装順序

1. **モジュール初期化**: `go mod init github.com/tkdn/gqlgen-subscription/backend`、`go get github.com/99designs/gqlgen@v0.17.93`
2. **gqlgen init**: `go run github.com/99designs/gqlgen init` で雛形生成
3. **擬似認証ミドルウェア** (`internal/userctx/userctx.go`): `net/http`ミドルウェアで固定UserIDを`context.Context`に注入する関数と、ctxからUserIDを取り出すヘルパー関数を用意。「本来はここでトークン検証」の趣旨コメントを残す
4. **スキーマ定義** (`graph/schema.graphqls`): 上記確定スキーマに書き換え、gqlgen initのサンプル（todos等）を削除
5. **コード生成**: `go run github.com/99designs/gqlgen generate`
6. **Redisクライアント** (`internal/redisclient/redisclient.go`): `go-redis/v9`を使い、環境変数`REDIS_ADDR`（デフォルト`localhost:6379`）からクライアント生成
7. **jobstore実装** (`internal/jobstore/store.go`):
   ```go
   type Store interface {
       Create(ctx context.Context, name string) (*model.Job, error)
       UpdateStatus(ctx context.Context, name string, status model.JobState) (*model.Job, error)
       List(ctx context.Context) ([]*model.Job, error) // ctxのUserIDを使う
   }
   ```
   `List`内で読み取り時ガベージコレクション（Setメンバーの実体確認と`SREM`）を実施。Create/UpdateStatus成功後にRedis Publish（`job:updates:<userID>`）も同メソッド内で行う
8. **Hub実装** (`internal/pubsub/hub.go`): UserIDごとに遅延Subscribe、fan-out、誰もいなくなったら接続破棄
9. **resolver実装** (`graph/schema.resolvers.go`): `Query.jobs`、`Mutation.createJob`/`updateJobStatus`はjobstoreを呼ぶだけ。`Subscription.jobStatuses`はHubへの登録/解除 + 初期スナップショット配信 + `ctx.Done()`監視（recipesの`defer close(ch)`パターン踏襲）
10. **server.go**: `srv.AddTransport(transport.SSE{...})`を最初に追加、擬似認証ミドルウェアをHTTPハンドラチェーンに組み込み、`Hub`をDIし、`signal.NotifyContext`でgraceful shutdown
11. **テスト**: jobstore/hub統合テスト（miniredis）→ resolver単体テスト → e2e SSEテストの順で書く

## フロントエンド実装順序

1. `pnpm dlx @angular/cli@latest new frontend --package-manager=pnpm`（standalone構成）
2. `pnpm add graphql-sse`
3. 型定義 (`models/job.model.ts`)
4. `app.config.ts`で`provideHttpClient(withFetch())`を設定
5. Query/Mutationサービス (`graphql/graphql-client.service.ts`): `HttpClient`で`/query`にPOST、`firstValueFrom`でPromise化
6. Subscriptionサービス (`graphql/graphql-subscription.service.ts`): `graphql-sse`の`createClient`をラップし`Observable<Job[]>`を返す
7. `job-board`コンポーネント: ジョブ名入力欄+作成ボタンで`createJob` mutationを呼び出し、一覧表示された各ジョブに`JobState`各値へのステータス変更ボタンを並べて`updateJobStatus` mutationを呼び出せるようにする（検証目的でどの状態にも自由に遷移可能）。`AsyncPipe`でsubscription結果を一覧表示
8. テスト (`job-board.spec.ts`): `@testing-library/angular`の`render()` + `msw`でQuery/Mutationをモック、subscriptionサービスはスタブに差し替え

## 開発サイクル

上記の「バックエンド実装順序」「フロントエンド実装順序」の各ステップを実装単位とする。1ステップ分のコードを書いたら、以下のサイクルを1ステップごとに繰り返す。

1. 該当ステップの実装を行う
2. `crit`を起動し、そのステップの差分についてレビューを受ける
3. レビューコメントに対応する（必要なら修正して再度`crit`でレビュー）
4. 承認されたら、そのステップ単位で`/commit`スキルを使ってコミットする
5. 次のステップに進み、1に戻る

ステップをまたいで複数のステップ分をまとめて実装・コミットしない。1ステップ＝1コミット（＋そのレビューサイクル）を基本単位とする。

## docker-compose.yml

Redisのみコンテナ化する（backend/frontendは開発ループ優先でローカル直接実行）。
```yaml
services:
  redis:
    image: redis:7-alpine
    ports:
      - "6379:6379"
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 5s
      timeout: 3s
      retries: 5
```

## 動作確認手順

1. `docker compose up -d redis` → `docker compose ps`でヘルスチェック確認
2. `cd backend && go run server.go` でサーバー起動
3. curlでmutation確認: `curl -s http://localhost:8080/query -H 'content-type: application/json' --data '{"query":"mutation { createJob(name: \"job-1\") { name status } }"}'`
4. curlでsubscription確認: `curl -N --request POST --url http://localhost:8080/query --data '{"query":"subscription { jobStatuses { name status } }"}' -H "accept: text/event-stream" -H 'content-type: application/json'` を張ったまま、別ターミナルで`updateJobStatus`を叩き、`event: next`が流れることを確認
5. Redis実データ確認: `docker compose exec redis redis-cli SMEMBERS user:<userID>:jobs`、`HGETALL job:<userID>:job-1`、`TTL job:<userID>:job-1`
6. `cd frontend && pnpm start` でブラウザ確認（`http://localhost:4200`）、DevTools NetworkタブでEventStream継続を確認
7. `cd backend && go test ./...`、`cd frontend && pnpm test`
