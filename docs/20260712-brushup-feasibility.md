# ブラッシュアップ3案の実現可能性見立て

[architecture.md](./architecture.md) §7時点（SQS完了通知フロー実装後）の構成を前提に、次の3案の実現可能性を評価したメモ。

1. サービスA→Bの依頼をSQSではなくHTTPにする
2. サービスBの完了はSQSのまま、SQSトリガーのLambdaがサービスAのジョブ関連データを更新する
3. Pub/SubをRedisではなくPostgreSQL NOTIFY/LISTENにする（今後DBとしてPostgreSQLを使う予定があるため）

## 総評

3点とも実現可能である。ただし性質が異なる。(2)と(3)は組み合わせると現行設計より構造的に良くなる可能性があるのに対し、(1)は「何を失うか」を明示した上で採否を決めるべき変更である。

## 1. サービスA→BをHTTPにする

**実装コストは3案中最小。** `graph.JobDispatcher` インターフェース（[`backend/graph/resolver.go`](../backend/graph/resolver.go)）が既に切られており、`sqsdispatch.Dispatcher` と差し替え可能な `httpdispatch` 実装と、workersim側のHTTPエンドポイント追加で済む。

ただし、この変更は「輸送手段の交換」ではなく**障害モデルの交換**である。

- 現行では B が停止していても `createJob` は成功する（キューが吸収する）。HTTP化すると `createJob` の成否が B の可用性に直結する。mutationをエラーにするか、A側にリトライ/Outboxを持つかの選択が新たに発生する。
- B が受理即応答（202 Accepted パターン）で処理を非同期化するなら、B の内部にキュー相当の仕組みが必要になる。つまり**キューは消えるのではなく B の内側に移動する**。
- HTTPリトライには冪等性が要るが、これは `job_id`（UUIDv7）が冪等キーとしてそのまま使える。

この構成が正当化される条件は「B が常時起動のサービスとして LB の背後にあり、受理は軽量で、処理のバックログ管理を B 自身が持つ」場合である。依頼＝コマンドはHTTP、完了＝イベントはキュー、という非対称は一般的なパターンとして成立する。逆に言えば、本リポジトリの検証目的が「キューによるバッファリングの体感」を含むなら、(1)はそれを検証範囲から外す変更になる。

## 2. 完了キュー→Lambdaトリガーでジョブデータ更新

**実AWS上では標準構成（SQS event source mapping）であり実現可能。** 本リポジトリで成立する決定的な理由は、**Pub/Subが既にRedisへ外部化されている**ことにある。Lambdaが `HSET` + `PUBLISH`（Postgres移行後は `UPDATE` + `NOTIFY`）を実行すれば、通知は稼働中のAPIプロセスの `pubsub.Hub` に届き、SSE配信まで連動する。architecture.md §5 で懸念されていた「インプロセスのイベントバスでは届かない」問題は、現行実装では既に存在しない。

留意点は3つ。

- **フレーミングの問題**: このLambdaを「第3のサービス」と捉えるとAのデータストアへの直接書き込み（shared database結合）になる。「サービスAのConsumerのデプロイ形態がFaaSに変わっただけ」と捉え、同一Goモジュール内で `jobstore` パッケージを共有し `cmd/lambda` エントリポイントを生やすのが素直である。現在の `consumer.Run` のループ内ロジックをメッセージ単位の純粋関数に抽出すれば、ローカルでは現行のポーリングループ、AWS上ではLambdaハンドラ、と同一関数を両方から呼べる。
- **冪等性が「先送り」から「前提」に格上げされる**: 現行consumerは単一goroutineの逐次処理だが、Lambdaは同時実行数が動的に増えるため、at-least-once配信の重複が顕在化しやすくなる。architecture.md §8で先送りした冪等化（条件付きUPDATE等）は(2)採用時の前提条件になる。
- **ローカル検証の可否が未確認**: [kumo](https://github.com/sivchari/kumo)はLambdaを対応サービスに挙げているが、event source mapping（SQS→Lambda起動のデータプレーン）まで実際に動くかはREADMEからは確認できなかった。コントロールプレーンAPIのモックのみの可能性がある。**ここが(2)の最初の検証項目である。** 動かない場合も、前述の「ハンドラ関数抽出」により、ローカルe2eは現行ループで担保しつつLambda形態はコードの形だけ検証する、という退避策が成り立つ。

## 3. Redis Pub/Sub→PostgreSQL NOTIFY/LISTEN

**3案の中で最も採用根拠が強い。** DBをPostgreSQLにする予定が既にあるなら、通知基盤を同居させることで現行設計の実際の弱点を1つ解消できる。すなわち、現行の `jobstore.save`（[`backend/jobstore/store.go`](../backend/jobstore/store.go)）では `HSET` と `PUBLISH` が別コマンドであり、間でプロセスが落ちると「DBは更新されたが通知が飛ばない」窓がある。Postgresでは `UPDATE` と `NOTIFY` を**同一トランザクション**に入れられ、通知はコミット成功時のみ発火する。「DBが正本、Publishはトリガーのみ」という現行の設計思想（architecture.md §4）に対して、Redisよりも忠実な実装になる。

現行設計との相性も良い。ペイロードを運ばず `List` でスナップショットを取り直す方式のため、NOTIFYのペイロード上限（8000バイト）は無関係であり、通知が非永続である点もRedis Pub/Subと同等（切断時はQuery再取得で回復、という既存の役割分担がそのまま効く）。

ただし移植には設計変更が1つ要る。現行 `pubsub.Hub` は**アクティブユーザーごとに専用のRedis Subscribe接続を張る**構造だが、これをそのままPostgresに写すと接続数が `max_connections` を圧迫する。正しい形は「単一のLISTEN接続で単一チャンネル（例: `job_updates`）を聴き、ペイロードに `userID` を載せてHub内でデマルチプレクスする」構造である。これはむしろ現行より接続効率の良い簡素化になる（userIDをチャンネル名でなくペイロードに置くことで、識別子63バイト制限の問題も回避できる）。

本番スケールでの本質的な注意点は1つ: **LISTENはセッションレベルの機能であり、transaction poolingを行うプロキシ（pgbouncer等）と両立しない**。将来コネクションプーリング層を挟む段になったら、LISTEN用接続だけプロキシを迂回させる等の分離が必要になる。これは今のスコープでは顕在化しないが、採用時にメモしておくべき制約である。

## 3案の合成と検証順

3案を全て採ると次の形になる。

```
A ──HTTP──▶ B ──SQS──▶ Lambda ──UPDATE+NOTIFY(単一tx)──▶ PostgreSQL
                                                              │ LISTEN
Client ◀──SSE/Subscription── A ◀──────────────────────────────┘
```

DBが「正本」と「ファンアウト起点」を兼ね、Aは非同期入力を一切SQSから受けなくなる。整合性の面では現行より強い。代償は、リアルタイム配信の可用性がDBの可用性に一点集中すること、完了1件ごとにDBラウンドトリップが必ず入ることだが、この検証規模では問題にならない。

検証順は **(3)→(2)→(1)** を推す。

1. **(3) PostgreSQL NOTIFY/LISTEN**: DB移行予定と重なり構造的な学びが最大。
2. **(2) Lambdaトリガー**: (3)のトランザクショナルNOTIFYと組んだときに最も引き立つ。kumoのevent source mapping対応確認という不確実性を先に潰せる。冪等性実装が前提条件。
3. **(1) HTTP依頼**: 独立性が高く差し替えも小さいので最後でよい。
