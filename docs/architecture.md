# ジョブ完了通知の設計指針（検証メモ）

エンドユーザからのリクエストを受け付けるサービス（本リポジトリではバックエンド一式、以下「サービスA」）が、非同期ワーカー（以下「サービスB」）に処理を依頼し、その完了をエンドユーザにリアルタイムで伝える、というクラウドネイティブなアーキテクチャについて壁打ちした内容を整理したもの。

**このドキュメントは確定した設計方針ではなく、技術検証のための検討メモである。** 本リポジトリ自体もGraphQL Subscription配信の疎通検証が目的であり（[README](../README.md)参照）、ここに書かれている内容も「こう考えると整理しやすい」という仮説・叩き台として扱う。実装で確定した判断は`plan/`配下の各メモを優先する。

前半はクラウドネイティブなジョブ完了通知について検討した一般的な考え方の整理、後半は本リポジトリの現在の実装がその考え方とどう関係するかの覚書。

## 1. 全体の考え方

```
Client
  │
  ▼
Service A ──enqueue──▶ Job Queue ──▶ Service B (Worker)
  ▲                                        │
  └───────────── completion notice ────────┘
```

「Bの完了通知をAがどう受け取るか」は、レスポンスではなく**イベント**として扱うと整理しやすい、というのが壁打ちでの結論だった。Bが「処理が終わった」という事実だけを発行し、Aを含む必要な購読者がそれを購読する形にすれば、通知先が増えてもBを変更せずに済む、という考え方。

このとき、AがBに投入したジョブと完了イベントを紐付けるために、一意な `job_id`（相関ID）をAが採番し、ジョブメッセージ・完了イベントの両方に一貫して載せる、というのがイベント駆動アーキテクチャで一般的に言われる考え方。

## 2. サービスAの役割：業務状態のオーケストレーター（という考え方）

各工程（B, C, D...）が自動的に連鎖するワークフローではなく、**次の工程に進むかどうかがエンドユーザの意思に依存する**ケースでは、Step Functions のような自動遷移オーケストレーションはオーバースペックになりやすい、という見立てだった。

そのケースでは、サービスAがジョブの業務状態（`job_id`, `status`, `step` 等）を自分のデータストアに保持する「プロセスマネージャー／オーケストレーター」として振る舞う案が出た。サービスBは業務フロー（どの工程か）を知らず、単一責務の処理を実行して結果を返すだけでよい、という整理。

壁打ちで出た判断基準（叩き台）：

| 要件 | 適した構成 |
|---|---|
| B→C→Dが自動連鎖し、リトライ・補償処理もAWSに任せたい | Step Functions 等のワークフローエンジン |
| 各工程はユーザ操作起点で独立して始まり、Aが業務状態を管理すればよい | Aが状態を持つプロセスマネージャー方式 |

## 3. 完了通知の配送経路：SQS vs EventBridge（AWS想定時の壁打ち）

サービスBは現時点で実装されておらず外部化もしていないが、仮にAWS上でジョブキュー＋ワーカーとして分離するとしたらどうなるか、という仮定の話として検討した内容。

- **SQS = 「この完了通知を誰か1人（=A）が確実に処理する」処理責任モデル**という整理。リトライ、バックプレッシャー（大量完了時にキューで吸収）に強いとされる。完了を必要とするのがAだけなら候補になりそう。
- **EventBridge = 「こういう出来事が起きた」という事実のファンアウトモデル**という整理。複数購読者（通知、監査、分析など）やイベント内容によるルーティングが必要になった場合の候補。

壁打ちで出た判断基準（叩き台）：

| 要件 | 候補 |
|---|---|
| Aだけが完了を知ればよい | SQS Completion Queue |
| 将来ほかのサービスも完了イベントを利用する可能性がある | EventBridge |

最初はSQS Completion Queueに寄せておき、購読者が増えた時点でEventBridgeを検討する、という進め方が案として出た。あくまで仮定の話であり、現時点で採否を決めたものではない。

## 4. リアルタイム配信（SSE / GraphQL Subscription）とポーリングの排除

エンドユーザへの状態配信を GraphQL Subscription（本リポジトリ）や HTTP SSE で行う場合、**DBを定期ポーリングしてクライアントに流す設計は避けたい**、というのが壁打ちでの見立て。ポーリングは、そもそもPush型の接続を使う利点を損なうため。

「状態が変化した瞬間にDB更新とクライアントへの配信を連動させる」形が理想として挙がった：

```
Service B ──completion event──▶ Service A
                                    │
                          ┌─────────┴─────────┐
                          ▼                   ▼
                       DB更新            クライアントへpush
                    (正本/Source of Truth)
```

DBを正本（Source of Truth）として持ち、Subscription/SSE接続が切れたクライアントは再接続時に通常のQuery（本リポジトリでは `Query.jobs`）で最新状態を取得する、という役割分担が考えられる。ポーリングが妥当になりそうなのは、Subscriptionを使わない・更新頻度が極めて低いなど、Push型を採用しない場合くらいではないか、という議論。

## 5. サービス内部構造：プロセス分割は責務ではなくスケール要件で決める、という考え方

「サービス」と「プロセス」は同一ではない、という前提のもと、GraphQL API（Query/Mutation/Subscription）・完了通知の受信（Consumer）は、どちらも「ジョブ状態を管理する」という同じ責務の異なる入口に過ぎないなら、同一プロセスに同居させてよいのではないか、という考え方が出た。

```
Service A (1プロセス)
 ├─ GraphQL API (Query/Mutation/Subscription)
 └─ Completion Consumer（Bからの通知を受ける入口）
```

分割を検討すべきタイミングは「ConsumerだけCPU/IOを大量に使う」「Consumerだけ別スケールさせたい」など、スケール特性が異なることが判明したときで十分ではないか、というのが壁打ちでの結論。最初から分けておく必然性はない、という整理。

分割する場合は、Consumerが受けた通知をGraphQL/SSE側に伝える経路（インプロセスのイベントバスでは届かない）を、Redis Pub/Sub や EventBridge のようなプロセス間の通知基盤に外部化する必要が出てくる、という見通し。この境界を最初から抽象化しておくと差し替えやすいのではないか、というアイデアが壁打ちの中で出た。ChatGPTとのやりとりで示されたコード片：

```go
type NotificationPublisher interface {
    PublishJobUpdated(ctx context.Context, jobID string)
}
```

開発初期は `InMemoryPublisher`（同一プロセスで完結）を使い、ECSで水平スケールが必要になったら実装だけ `RedisPublisher` や `EventBridgePublisher` に差し替える、という案。アプリケーションのユースケースやドメインロジックには一切手を入れずに、通知基盤だけを外部化できる、という点がこのアイデアの肝。

## 6. 水平スケール時に踏みそうな課題

サービスAをECS等で複数タスクに水平スケールする場合、次の2点が典型的な落とし穴になりそう、という想定。

### (1) 通知を受けたタスクと、配信先の接続を持つタスクが一致しない

```
          SQS / EventBridge
              │
        ┌─────┴─────┐
        ▼           ▼
     Task A       Task B
   (通知を受信)  (SSE接続を保持)
```

Task Aが完了通知を受けてDBを更新しても、そのユーザーのSSE接続はTask Bにある可能性がある。インプロセスのイベントバスだけでは届かないため、Redis Pub/Sub や EventBridge のようなタスク間の通知共有が要りそう、という想定。本リポジトリでは、この問題自体は単一プロセス構成のため今のところ発生しない（後述）。

### (2) Consumerのタスク数増加が下流リソースを圧迫するかもしれない

「ECS Auto Scalingでタスクが増える → 全タスクが同時にDBを更新しに行く → DBコネクション枯渇」という事故が起こりうる、という懸念から出た対策案：

- SQSは `WaitTimeSeconds` を使ったロングポーリングにし、ショートポーリングでAPI呼び出し回数を無駄に増やさない。
- タスク数だけでなく、タスク内の並列数（goroutine数など）も制御し、「タスク数 × 並列数 × 下流API呼び出し数」の総量を意識する。
- SQSはat-least-once配信のため、Consumer側の更新処理は冪等にする（例: `WHERE status != 'completed'` 条件付きUPDATE、または処理済みイベントIDを記録するInboxテーブル）。

## 7. 本リポジトリでの現在の実装との関係（覚書）

現在の実装は「サービスAの内部構造」の検証に主眼があり、サービスBは実装されていない（ジョブの状態変化は検証用mutationで直接発生させる）。上記で整理した考え方と、現在の実装がどう関係しそうかのメモ。

| 考え方 | 現在の実装との関係 |
|---|---|
| 完了通知はイベントとして扱う | `jobstore.Store.save` が更新のたびに `job:updates:<userID>` チャンネルへPublishしている（[`backend/jobstore/store.go`](../backend/jobstore/store.go)）。検証用mutationからの状態変化をイベントとして扱う形になっている |
| 相関ID | ジョブ名 (`name`) をキー・チャンネル名の両方に用いて紐付けている |
| DBが正本、配信はイベント駆動 | Redis Hash (`job:<userID>:<name>`) が正本。Publishはトリガーのみ（`chan struct{}`）で、受信側は `List` を呼び直して全件スナップショットを取得する方式（[`backend/pubsub/hub.go`](../backend/pubsub/hub.go) 参照）。ポーリングはしていない |
| プロセス分割はスケール要件で決める | GraphQL API・Redis Subscribe（Consumer相当）は同一プロセス（`backend/cmd/main.go`）に同居している。水平スケールは検証範囲外 |
| 冪等性 | 検証用mutationは都度上書き（`HSet`）で完結するため、現状は問題にならない |
